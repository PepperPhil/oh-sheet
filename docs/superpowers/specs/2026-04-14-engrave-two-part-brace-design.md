# Engrave Two-Part Braced Grand Staff Design

**Date**: 2026-04-14
**Status**: Approved

## Overview

Switch `engrave` from emitting a single `<part>` with `<staves>2</staves>` to emitting two `<part>` elements joined by a `<part-group><group-symbol>brace</group-symbol></part-group>`. This is a data-layer fix to the MusicXML structure; it does not add a new tool, renderer, or dependency.

## Problem

Rendered PDFs from the `engrave` stage currently show both staves using a treble clef, with left-hand notes drawn as massive ledger-line columns below the staff and the time signature placed in an incorrect position. The failure was reproduced on Bob Marley job `f255d56b4243` (YouTube `oFRbZJXjWIA`).

The score data is correct — both hands split cleanly by pitch (RH pitches 61–88, 598 notes; LH pitches 40–59, 503 notes) and the emitted MusicXML declares both clefs correctly (`<clef number="1">G</clef>` / `<clef number="2">F</clef>`). The bug is in `musicxml2ly`, the MusicXML → LilyPond bridge tool: it does not reliably honor music21's one-part/two-staves encoding, so LilyPond collapses both staves into a single treble-clef rendering.

MuseScore handles the current encoding correctly. LilyPond is our preferred renderer because it's smaller, faster, and easier to ship on Cloud Run, so the fix targets the MusicXML encoding rather than the renderer choice.

## Approach

MusicXML supports two idioms for piano grand staff:

- **One-part / multi-staff** (current): `<part>` with `<staves>2</staves>`, each `<note>` carries `<staff>1</staff>` or `<staff>2</staff>`. Generated by `music21.stream.PartStaff`.
- **Two-part / braced** (new): two `<part>` elements, joined in `<part-list>` by `<part-group type="start" number="1"><group-symbol>brace</group-symbol></part-group>` ... `<part-group type="stop" number="1"/>`. Notes carry no `<staff>` tag. Generated by `music21.stream.Part` × 2 + `music21.layout.StaffGroup([...], symbol='brace')`.

Both are valid. The two-part encoding is handled identically by LilyPond (via `musicxml2ly`), MuseScore, Verovio, and OSMD — it avoids the bridge bug.

## Implementation

### Swap in `backend/services/engrave.py`

Current structure (around lines 504–650):

```python
# two PartStaff objects inside a single Part, bound by StaffGroup brace
for hand_name, notes, clef in (
    ("Right Hand", score.right_hand, music21.clef.TrebleClef()),
    ("Left Hand", score.left_hand, music21.clef.BassClef()),
):
    part_staff = music21.stream.PartStaff()
    part_staff.append(clef)
    ...
```

New structure:

```python
# two Part objects, braced by StaffGroup
rh_part = music21.stream.Part(id="RH")
rh_part.append(music21.clef.TrebleClef())
# key sig, time sig, tempo mark, RH notes by measure

lh_part = music21.stream.Part(id="LH")
lh_part.append(music21.clef.BassClef())
# same key/time sig, LH notes by measure

out_score = music21.stream.Score()
out_score.insert(0, rh_part)
out_score.insert(0, lh_part)
out_score.insert(0, music21.layout.StaffGroup(
    [rh_part, lh_part], symbol="brace", barTogether=True
))
```

`PartStaff` is gone from this path. The surrounding measure-building, note emission, tie handling, pedal markings, and `_MAX_BEATS` guard are unchanged.

### Downstream ripple

1. **Voice remapping** (`_remap_voices_per_staff`, engrave.py:719–785). Today it reads `<staff>` tags within one `<part>` and compresses voice numbers per staff. Under two-part encoding, there is no `<staff>` tag — each `<part>` owns its voices. Rename to `_remap_voices_per_part` and iterate over `<part>` elements, collecting and compressing voice numbers inside each. If voices already stay in `{1, 2}` per hand (expected), the function may be a no-op and can be removed; keep it for defense-in-depth until verified.
2. **Tie-stop voice fixup** (engrave.py:787–820). Today keyed on `(pitch, staff)` within one part. Change to iterate per `<part>` independently, keyed on `pitch` alone. Tied notes do not cross hands, so this is a strict simplification.
3. **Chord symbols** (engrave.py:614). Today attached to the combined stream. Attach to `rh_part` only — standard convention, same visual placement above the top staff.
4. **Score-level metadata** (title, composer). Apply `music21.metadata.Metadata` to the `Score` object rather than the combined stream.

### Contract impact

None. `shared/shared/contracts.py` is untouched. `EngravedOutput` shape is unchanged — `musicxml_uri`, `pdf_uri`, `midi_uri` still produced.

### Behavioral change to flag

External consumers of the `musicxml` artifact (if any) that parse the file directly will see a restructured XML tree: two `<part>` elements instead of one. The rendered PDF is the primary consumer and is unaffected visually — if anything, it's now correct. No known internal consumer parses MusicXML shape.

## Testing

### Unit tests (`tests/test_engrave_quality.py` and siblings)

1. **Structure assertion**: after engrave, the emitted MusicXML has exactly two `<part>` elements, one `<part-group>` with `<group-symbol>brace</group-symbol>`, a treble clef on part 1, a bass clef on part 2, and no `<staves>` element.
2. **Note counts**: notes under part 1 equal `len(score.right_hand)`; notes under part 2 equal `len(score.left_hand)`.
3. **Regression test**: a fixture `PianoScore` with pitch ranges matching the Bob Marley failure (RH C4–E6, LH E2–B3) renders to MusicXML with both clefs correctly assigned.
4. **MuseScore-corruption invariants**: keep the existing tests that verify the renderer does not see tie/voice corruption. These should still pass under two-part encoding — they're about per-note well-formedness, not staff structure.

### Render-level verification (manual, one-shot)

Documented in the implementation plan, not automated:

- Re-run job `f255d56b4243` after the fix and confirm the PDF shows a proper grand staff: treble clef on top, bass clef on bottom, 4/4 time signature aligned correctly, left-hand notes rendering in a normal range without a sea of ledger lines.
- Render one simpler pre-existing job to confirm no regression on cases that worked before.

### Type / lint

`make typecheck && make lint` stay clean. If `Part` vs `PartStaff` introduces a music21 type stub mismatch, cast or suppress narrowly at the call site.

## Rollout

Clean cutover. A single commit replaces the `PartStaff` path with the `Part + StaffGroup` path. No feature flag, no dual-path code, no fallback to the old encoder. Rationale: the current path is verifiably broken on at least one real job, test fixtures validate MusicXML shape, and feature-flagged XML encoders tend to linger indefinitely.

Post-merge: re-process the failing Bob Marley job end-to-end to confirm the fix lands in production.

## Out of Scope / Future Work

- **Mid-piece clef changes.** Real piano music occasionally uses two treble clefs or two bass clefs for sections where notes drift out of the normal range for the other clef. Supporting this requires emitting `<clef>` changes inside measures where the dominant pitch range drifts. This proposal keeps the fixed treble-RH / bass-LH default; clef-change sections remain a stretch goal to resolve once the base case is solid.
- **Alternative renderers** (Verovio, MusicXMLJS, etc.). Unnecessary once the MusicXML is renderer-agnostic — LilyPond and MuseScore both handle the new encoding correctly.
